# 01. Project Overview

## 1. 프로젝트 이름

**RAG Code Reviewer**

FastAPI 기반 Codebase + Official Docs RAG 코드 리뷰 시스템입니다.

---

## 2. 프로젝트 목적

이 프로젝트의 목적은 개발자가 자신의 코드베이스와 공식문서를 함께 검색하여, 코드의 동작 방식과 문제 가능성을 더 빠르게 파악할 수 있도록 돕는 것입니다.

일반적인 LLM은 사용자의 프로젝트 내부 코드를 알지 못하고, 최신 공식문서와 다른 답변을 생성할 수 있습니다. 이 프로젝트는 RAG 구조를 사용하여 사용자의 코드와 공식문서를 검색한 뒤, 검색된 근거를 기반으로 답변을 생성합니다.

---

## 3. 해결하려는 문제

개발자는 코드 리뷰나 디버깅 과정에서 다음과 같은 문제를 자주 겪습니다.

1. 특정 기능이 코드의 어느 위치에서 처리되는지 찾기 어렵다.
2. 프레임워크 공식문서 기준으로 현재 코드가 적절한지 판단하기 어렵다.
3. 공식문서, 코드, 에러 로그를 따로 확인해야 해서 시간이 오래 걸린다.
4. LLM에게 질문해도 프로젝트 내부 코드 근거 없이 일반적인 답변만 받을 수 있다.
5. 리뷰 결과가 어떤 코드와 어떤 문서를 근거로 하는지 추적하기 어렵다.

이 프로젝트는 위 문제를 해결하기 위해 코드베이스와 공식문서를 각각 인덱싱하고, 질문마다 관련 근거를 검색하여 답변을 생성합니다.

---

## 4. 주요 사용자

초기 사용자는 다음과 같이 가정합니다.

- 백엔드 개발자
- FastAPI 학습자
- 코드 리뷰를 준비하는 개발자
- 프로젝트 온보딩이 필요한 개발자
- 공식문서 기준으로 코드 사용법을 검증하고 싶은 개발자

초기 버전은 개인 개발자의 로컬 사용을 기준으로 합니다.

---

## 5. 핵심 사용 시나리오

### 5.1 코드 흐름 설명

사용자가 다음과 같이 질문합니다.

```text
카메라 생성 API 흐름 설명해줘.
```

시스템은 관련 router, service, schema, model 코드를 검색한 뒤 API 흐름을 설명합니다.

---

### 5.2 공식문서 기준 코드 리뷰

사용자가 다음과 같이 질문합니다.

```text
이 JSONResponse 사용 방식 공식문서 기준으로 괜찮아?
```

시스템은 프로젝트 코드에서 JSONResponse 사용 위치를 검색하고, FastAPI 공식문서에서 Response 직접 반환 및 jsonable_encoder 관련 내용을 검색합니다. 이후 코드가 공식문서 기준으로 적절한지 판단합니다.

---

### 5.3 에러 처리 검토

사용자가 다음과 같이 질문합니다.

```text
이 코드에서 예외 처리 구조가 괜찮아?
```

시스템은 exception handler, service layer, error code 문서를 검색하여 예외 처리 일관성을 검토합니다.

---

## 6. 주요 기능 요약

| 기능 | 설명 |
|---|---|
| 코드 인덱싱 | 로컬 프로젝트의 코드 파일을 읽고 chunk 단위로 저장 |
| 공식문서 인덱싱 | Markdown 형식 공식문서를 chunk 단위로 저장 |
| 코드 검색 | 질문과 관련된 코드 chunk 검색 |
| 공식문서 검색 | 질문과 관련된 공식문서 chunk 검색 |
| RAG 리뷰 | 검색된 코드와 문서를 기반으로 LLM 답변 생성 |
| 근거 반환 | 관련 코드 위치와 문서 출처 반환 |
| 요청 로깅 | request_id, latency, status_code 기록 |
| RAG 로깅 | 검색된 chunk 수, 모델명, 처리 시간 기록 |
| 에러 응답 표준화 | 공통 에러 응답 형식 제공 |

---

## 7. 초기 MVP 범위

초기 MVP는 로컬 환경에서 다음 기능을 제공하는 것을 목표로 합니다.

- FastAPI 서버 실행
- 코드 폴더 인덱싱
- Markdown 공식문서 인덱싱
- 코드 + 공식문서 검색
- LLM 기반 리뷰 결과 생성
- request_id 기반 구조화 로그 출력
- Swagger UI를 통한 API 테스트

---

## 8. 초기 범위에서 제외하는 것

다음 기능은 초기 MVP에서 제외합니다.

- 사용자 로그인 및 권한 관리
- GitHub PR 자동 리뷰
- 클라우드 배포
- 다중 사용자 프로젝트 관리
- 실시간 협업 기능

로컬 개인 사용을 편하게 하기 위한 최소한의 프론트엔드(Streamlit UI, #32)는 MVP 범위에 포함한다. Swagger UI로 매번 JSON을 직접 작성하지 않고도 프로젝트 등록/인덱싱/리뷰/히스토리 조회를 할 수 있다.

해당 기능들은 RAG 핵심 파이프라인이 안정화된 이후 확장 대상으로 둡니다.

---

## 9. 성공 기준

초기 MVP의 성공 기준은 다음과 같습니다.

1. 로컬 코드 프로젝트를 인덱싱할 수 있다.
2. 공식문서 Markdown 파일을 인덱싱할 수 있다.
3. 질문에 대해 관련 코드 chunk와 공식문서 chunk가 검색된다.
4. LLM 답변에 코드 위치와 공식문서 근거가 포함된다.
5. 모든 API 요청에 request_id가 부여된다.
6. RAG 실행 과정이 로그로 남는다.
7. Swagger UI에서 주요 API를 테스트할 수 있다.
